This is another common layout error you could run into while creating a UI in your Flutter app.

What does the error look like?

How might you run into this error?

When a ListView (or other types of scrollable widgets like as GridView) is put inside a Column, the issue frequently occurs. Unless it is bound by its parent widget, a ListView uses all of the vertical space available to it. A Column, on the other hand, does not impose any height constraints on its children by default. The combination of the two actions results in the failure to determine the ListViewâ€™s size.

              title: Text('Map'),

            ListTile(
              leading: Icon(Icons.subway),
              title: Text('Subway'),
            ),
          ],
        ),
      ],
    ),

How to fix it?

To resolve this issue, define the height of the ListView. Wrap it with an Expanded widget to make it as tall as the remaining space in the Column (see the example below). Otherwise, use a SizedBox widget to indicate an absolute height or a Flexible widget to specify a relative height.

  return Center(
    child: Column(
      children: <Widget>[
        Text('Header'),
        Expanded(

            children: <Widget>[

5. setState called duringÂ build

The build method in your Flutter code is not a good place to call setState either directly or indirectly.

What does the error lookÂ like?

When the error occurs, the following message gets displayed in the console:

The following assertion was thrown building DialogPage(dirty, dependencies: 
[_InheritedTheme, _LocalizationsScope-[GlobalKey#59a8e]], 
state: _DialogPageState#f121e):
setState() or markNeedsBuild() called during build.
This Overlay widget cannot be marked as needing to build because the framework 
is already in the process of building widgets.
(Additional lines of this message omitted)

How might you run into theÂ error?

This problem generally arises when the setState function is invoked within the build method.

This problem frequently arises when attempting to initiate a Dialog from within the build procedure. This is frequently prompted by the desire to present information to the user right away, but setState should never be called from a build function.

Below is a snippet that seems to be a common culprit of this error:

Widget build(BuildContext context) {
  // Don't do this.
  showDialog(
      context: context,
      builder: (BuildContext context) {
        return AlertDialog(
          title: Text("Alert Dialog"),
        );
      });
  return Center(
    child: Column(
      children: <Widget>[
        Text('Show Material Dialog'),
      ],
    ),
  );
}

The explicit call to setState is not visible, but it is invoked by showDialog. The build function is not the appropriate location to call showDialog since the framework can call build at any time, such as during an animation.

How to fixÂ it?

One solution is to use the Navigator API to activate the dialogue as a route. The sample below has two pages. The second page contains a dialogue that will be given upon entry. When a user requests the second page by clicking a button on the first page, the Navigator sends two routes in a row, one for the second page and one for the dialogue.

class FirstScreen extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(
        title: Text('First Screen'),
      ),
      body: Center(
        child: ElevatedButton(
          child: Text('Launch screen'),
          onPressed: () {
            // Navigate to the second screen using a named route.
            Navigator.pushNamed(context, '/second');
            // Immediately show a dialog upon loading the second screen.
            Navigator.push(
              context,
              PageRouteBuilder(
                barrierDismissible: true,
                opaque: false,
                pageBuilder: (_, anim1, anim2) => MyDialog(),
              ),
            );
          },
        ),
      ),
    );
  }
}
